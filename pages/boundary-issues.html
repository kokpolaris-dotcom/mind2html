01 底座的无代码引擎 Agent 研发为何受限于无代码底座？ Code vs NoCode
困境 Dilemma

目前做智能体的开发，受限于"底座"的开发进度。

底座没就绪，业务侧 Agent 就只能排队等，研发产出无法转化为上线价值。

讨论 Discussion

为什么会受限于底座？团队认知是："所有 Agent 必须上无代码 Agent 引擎底座"。

底座是不是只能是无代码的？

一堆最擅长代码的 Agent 研发，却被困在无代码引擎上 —— 这个是否合理？

建议 Proposal

不合理。底座应是 代码 + 无代码双引擎 驱动。

  1. 底座不成熟时，先通过代码快速应对核心场景价值交付，并沉淀出底座真正要解决的业务 Agent 核心问题
  2. 解决特别复杂、开发门槛高的 Agent 开发问题，比如基于项目画布的多 Agent 协作高质量内容创作
双引擎同源：共用 Tools / RAG / 模型路由，PoC 跑通后可沉淀回生产底座
02 Agent 纳管边界与生态定位 权限控制归属决定底座生态 Tenant × Ecosystem
困境 Dilemma

因租户与权限问题，引发产品边界、生态定位的思考。

一个具体场景的权限归属疑问，牵出整个 Agent 体系的归属权与控制权问题。

讨论 Discussion

原问题：新媒体情报官的权限控制，是来自原多多系统，还是底座的权限体系？

由此延展到 Agent 的归属权、控制权与生态问题。

若 Agent 权限全来自多多而非底座，那么底座的生态逻辑何在？

建议 Proposal

不管官方自建或生态共建的 Agent，本质统一由底座统一纳管。

  1. 纳管对象：所有 Agent（自建 / 生态 / 三方）统一接入底座注册中心
  2. 权限收口：租户、角色、权限由底座统一颁发，多多等业务系统只做业务态数据权限
  3. 生态闭环：底座 = Agent 的生命周期 + 权限 + 计量 + 飞轮的唯一入口
底座即生态：Agent 的"户籍"必须在底座，业务系统只是"使用方"

02 — Executive Summary

执行摘要

产品侧

双层级产品架构：底座平台(L1)+ 官方专家分身(L2)+ 用户自定义分身(L3)，形成"AaaS + Agent Foundry"双商业模式。 从营销域切入，逐步扩展到经营域乃至全经营链路。

技术侧

五大核心模块：编排引擎 + RAG 2.0 知识中台 + Skill/Tools 层 + Agent Builder + 数据飞轮，形成"感知→决策→执行→反馈→进化"完整闭环。

核心理念：把专家认知做成可规模化的数字分身

平台既自己造“数字员工”（官方分身 / AaaS），也开放“铸造厂”（Foundry / 客户自定义）——两者相互促进，形成双向飞轮。

03 — Industry & Pain Points

行业背景与核心痛点

专家能力稀缺且昂贵

文旅各经营环节都依赖"资深老师傅"——定制游产品经理、收益管理专家、运营操盘手。人才贵、少、难培养，人员流动直接影响业务质量。

经营链路长、协同复杂

从内容种草、产品设计、供应链协调、价格管理到用户运营，涉及大量长链路决策。人工模式信息传递慢、决策延迟、协同成本高。

人力模式难规模化

无论是代运营、定制游还是航司收益管理，依赖专家人力的模式边际成本高，服务能力有明显天花板。金牌专家产出无法无限复制。

Know-how 无法沉淀传承

大量经营 Know-how 存在于专家头脑中，没有被系统化记录、结构化、传承。人员一换，经验就流失，组织能力无法积累。

09 — AI Architecture Decisions (1/2)

AI 架构核心决策 · 上

①

Agent 编排引擎

问题域：文旅营销是长链路、多步骤、跨角色任务（选题→脚本→分发→复盘），非单轮 QA
▸

多 Agent 协作框架

Mastra

匹配：原生 Agent Network + Workflow，支持 DAG

理由：① TS 原生与 NestJS 无缝 ② 比 LangGraph 省 Python 跨语言层 ③ Workflow+Network 双模型对应"流水线"与"协作"

关键：不用 LangGraph —— 偏 Python 生态，跨语言增加运维成本

▸

Planner 架构

ReAct + Function Calling

匹配：需边推理边调工具（查热点→写脚本→查规则→调整）

理由：① ReAct 适合"思考-行动-观察"循环 ② 比 Plan-Execute 灵活，可中途调整 ③ 比 ToT 成本低

关键：不用纯 Function Calling —— 缺"反思"，长任务会跑偏

▸

State Management

Redis + Checkpoint

匹配：任务可能跨小时（等数据回流），需断点续跑。Redis 存活跃上下文，Checkpoint 落 PG 支持恢复

②

RAG 2.0 知识中台

问题域：大量专有名词 + 时效热点 + 多租户私有认知，纯向量检索会出错
▸

混合检索三路融合

向量 + BM25 + KG

匹配：① 景区 IP 名相似但不同 ② 营销话语语义相似 ③ 非遗/地名有层级关系

理由：向量负责语义召回，BM25 精确名词命中，KG 负责关系推理（"九寨沟"→"四川"→"山水类景区"）

关键：不用纯向量 —— "千岛湖"和"太平湖"向量相似但完全不同景区

▸

多租户知识隔离

Milvus Namespace

匹配：官方分身用行业库，自定义分身用"行业+私有"，严格隔离

理由：① Milvus 原生 partition/namespace ② 比应用层过滤更安全 ③ 支持租户级检索权重调优

▸

知识动态更新

ETL + 增量向量化

匹配：文旅热点 T+1 失效（某景区突然爆火）。抓取→清洗→Embedding→入库全自动，热点标签单独索引

10 — AI Architecture Decisions (2/2)

AI 架构核心决策 · 下

③

Skill / Tools 协议

问题：Agent 如何理解工具能力边界？

Skill 协议

MCP + FC 回退

理由：MCP 比 OpenAPI 更面向 LLM，含 when-to-use 语义；FC 回退兼容旧模型

关键：不用纯 OpenAPI —— 描述的是 HTTP 接口而非 Agent 可理解的能力

沙箱执行

gVisor

理由：用户自定义 Skill 含未知代码，内核级隔离比 Docker 更安全

④

Agent Builder 铸造厂

问题：如何让非技术运营专家造出可用 Agent？

认知注入

Few-shot + RAG

理由：运营专家不懂 Prompt 但擅长举例，上传优秀文案作 Few-shot + RAG 检索相似场景

关键：不用 Fine-tuning —— ToB 样本少（几十条），过拟合风险高

仿真评估

Replay + LLM-Judge

理由：历史数据回放 + LLM 评分（合规/调性/转化）

关键：LLM-Judge bias → 双模型交叉评分，分歧时人工复核

⑤

Data Flywheel 进化

问题：如何让 Agent持续变聪明？

反馈回流

DPO

理由：点赞/点踩天然成偏好对，DPO 不需 Reward Model，比 PPO 简单

关键：不用 PPO —— 需 RM + RL 训练循环，ToB 团队维护不起

A/B 评估

Bayesian

理由：ToB 流量小，Bayesian 在 10-100 样本下也能给显著性判断

飞轮闭环 反馈 → Skill 沉淀周期 < 1 周

12 — Summary

总结

不仅自己造"数字员工"，
还开放"铸造厂"。

把专家认知做成数字分身（AaaS），把铸造能力开放给行业（Foundry）——AI 时代不是卖软件，是卖认知 + 卖铸造认知的能力。

产品侧

营销域切入 → 经营域纵深 → 全经营链路。双层级架构覆盖 AaaS + Foundry 双边市场，从工具跃升到生态平台。

技术侧

五大核心模块形成完整闭环。多领域知识库架构支持高效扩展，数据飞轮构建持续进化的能力底座。

壁垒侧

五重壁垒层层叠加。Know-how 越沉淀越厚，飞轮越转越快，壁垒随时间递增。

## 一、执行摘要

Wanpix.ai 是面向文旅行业的**经营域 Know-how Agent 化平台**。核心理念是:把企业自身沉淀的 Know-how(专业经验、操作逻辑、决策方法论)转化为平台的核心竞争力,以 AI Agent 的形式规模化复制——既自己造车(官方分身),也开放造车能力(PaaS 底座)。

能力覆盖**营销域 + 经营域**两大板块:

- **营销域**:内容营销、热点捕捉、文案生成、平台分发、投放优化等
- **经营域**:旅行社定制游路线设计、航司收益管理、目的地运营、供应链协调等

本方案提出"**双层级产品架构 + 五大核心技术模块**"的整体设计:

- **产品侧**:底座平台(Layer 1)+ 官方专家分身(Layer 2)+ 用户自定义分身(Layer 3),形成"SaaS + PaaS"双商业模式。
- **技术侧**:编排引擎 + RAG 2.0 知识中台 + Skill/Tools 层 + Agent Builder + 数据飞轮,形成"感知 → 决策 → 执行 → 反馈 → 进化"的完整闭环。

该方案的核心壁垒在于:**把 Know-how 做成平台的核心竞争力,不仅自己造"车"(官方分身),还开放顶级"造车平台"(Harness 底座)**——这在 ToB 业务中是一个极具想象力和壁垒的商业模式。

## 二、项目背景与行业洞察

### 2.1 行业现状

文旅行业参与者可划分为三类,各自卡在经营链路的不同环节:

| 参与者 | 能力现状 | 核心短板 |
|---|---|---|
| 上游(景区、酒店、目的地) | 拥有优质物理产品 | 缺乏系统化经营能力:营销、运营、供应链 |
| 中游(旅行社、航司、渠道) | 拥有渠道与供应链 | 数字时代经营效率低、专业人才稀缺 |
| 传统代运营 / 服务商 | 能解决单点问题 | 人力模式无法规模化,质量参差不齐 |

行业痛点不局限于营销,而是贯穿整个经营链路——旅行社定制游的路线设计与报价、航司的收益管理与舱位调度、目的地的整体运营协调,都依赖稀缺的专家经验。

### 2.2 痛点分析

#### 痛点一:专家能力稀缺且昂贵

文旅行业各个经营环节都依赖"资深老师傅"——旅行社的定制游产品经理、航司的收益管理专家、目的地的运营操盘手。这些人才贵、少、难培养,且人员流动直接影响业务质量。

#### 痛点二:经营链路长、协同复杂

从内容种草、产品设计、供应链协调、价格管理到用户运营,文旅经营涉及大量长链路决策。人工模式下信息传递慢、决策延迟、协同成本高。

#### 痛点三:传统人力模式难规模化

无论是代运营、定制游还是航司收益管理,依赖专家人力的模式边际成本高,服务能力有明显天花板。金牌专家的产出无法无限复制,业务扩张受制于人才瓶颈。

#### 痛点四:Know-how 无法沉淀传承

大量经营 Know-how 存在于专家头脑中,没有被系统化记录、结构化、传承。人员一换,经验就流失,组织能力无法积累。

### 2.3 机会洞察

将各经营域的专家 Know-how,通过 AI Agent 技术转化为可规模化复制的数字能力,实现:

- **经验固化**:把人的隐性 Know-how 变成系统的显性能力
- **规模复制**:突破人力天花板,一个专家的能力服务一万个客户
- **持续进化**:数据飞轮让系统越来越聪明,不断自我迭代
- **开放赋能**:把沉淀的能力开放出去,形成平台生态

这是 AI Agent 技术成熟后,文旅经营行业出现的历史性机会窗口。

## 三、产品定位与价值主张

### 3.1 核心定位

> **Wanpix.ai 不是降本工具,而是文旅经营域 Know-how 的 Agent 化平台。**

核心理念:**把自己的 Know-how 做成平台的核心竞争力**。通过将专家经验、操作逻辑、决策方法论沉淀为可编排的 Agent 能力,平台既可以自己造车(做官方分身、做自营业务),也可以开放能力(做 PaaS、让客户自己造车)。

它解决的不是单点效率问题,而是**经营能力缺口**——将各经营域的深域专家经验数字化为可规模化复制的 Agent 能力,实现专家级能力的平权与规模化复制。

### 3.2 能力域覆盖

| 能力域 | 典型场景 | 专家 Know-how 举例 |
|---|---|---|
| **营销域** | 内容营销、热点运营、平台分发 | 爆款文案规律、流量转化逻辑、平台调性把握 |
| **经营域** | 旅行社定制游、航司收益管理 | 路线设计逻辑、动态定价模型、舱位调度策略 |
| **运营域** | 目的地运营、用户增长 | 活动策划 SOP、用户分层运营、生命周期管理 |
| **供应链域** | 资源采购、供应商协调 | 议价模型、履约调度、品控标准 |

> 营销域是切入点,经营域是纵深方向。底层理念完全一致:Know-how 沉淀 → Agent 封装 → 标准化交付 → 飞轮进化。

### 3.3 价值链定位

四类参与者在价值链上各司其职,Wanpix.ai 处在最稀缺的"能力基建"层:

| 参与者 | 提供的能力 | 本质 |
|---|---|---|
| 上游 & 中游 | 提供"物理世界的产品与服务" | 资源 / 服务 |
| 传统服务商(代运营/定制游) | 提供"人力驱动的专业服务" | 非标服务 |
| 纯工具平台 | 提供"生产效率工具与算力" | 工具 / 算力 |
| **Wanpix.ai** | **提供"经营域专家能力的数字化分身"** | **能力基建** |

### 3.4 三大核心价值主张

#### 价值一:Know-how 沉淀

把实战中积累的**隐性 Know-how**——什么样的定制游产品好卖、什么时间点调价转化率最高、如何设计路线组合套餐——提炼成结构化的策略模型与知识图谱,成为 Agent 决策的底层认知。

#### 价值二:Agent 封装

把"人脑经验"转化为"Agent 执行逻辑"。它不再只是生成文字,而是具备**经营思维**——像资深产品经理/收益管理专家一样自动分析数据、做出决策、执行动作。从"知道怎么做"到"系统自动做"。

#### 价值三:标准化交付

把依赖"金牌专家个人能力"的非标服务,变成可无限复制的**标准化数字服务**。企业等于以极低成本雇佣了一支顶尖的超级数字经营团队。

## 四、产品方案

### 4.1 核心理念

> **把 Know-how 做成平台的核心竞争力。**
>
> 平台既自己造车(官方分身 / 自营业务),也开放造车能力(PaaS / 客户自定义)。

这是一个双向飞轮的模式:

```
        自营业务 Know-how 沉淀
        ┌──────────────────────┐
        ▼                      │
   平台底座能力增强             │
        │                      │
        ▼                      │
   开放赋能吸引客户             │
        │                      │
        └── 客户新玩法反哺 ─────┘
```

一方面,自营业务(官方分身)沉淀的 Know-how 持续增强底座能力;另一方面,开放底座吸引客户构建自定义分身,客户产生的新玩法又反哺底座,形成双向增强的飞轮。

### 4.2 双层级产品架构

把"能力"变成可组装的"乐高积木"——商业天花板从 SaaS 工具跃升到 **PaaS 生态平台**。

```
┌─────────────────────────────────────────────────────┐
│  Layer 3 · 用户自定义分身  (生态化 · 长尾赋能)        │
│  客户调用底座能力,注入私有认知,构建专属 Agent         │
└───────────────────────▲─────────────────────────────┘
                        │ 依赖
┌───────────────────────┴─────────────────────────────┐
│  Layer 2 · 官方专家分身  (标准化交付 · 标杆示范)      │
│  开箱即用的标准化产品,解决行业 80% 通用需求           │
└───────────────────────▲─────────────────────────────┘
                        │ 基于
┌───────────────────────┴─────────────────────────────┐
│  Layer 1 · 底座平台  (能力原子化 · 行业基建)          │
│  Harness 技术:RAG 行业知识库 + Skill + Tools         │
└─────────────────────────────────────────────────────┘
```

#### Layer 1 — 底座平台(能力原子化)

文旅经营领域的"安卓系统",提供最底层的行业水电煤。Harness 技术(RAG / Skill / Tools)本质是把各经营域积累的"核心能力"进行结构化拆解与原子化封装——文旅经营领域的"操作系统"。

- **RAG 知识库**:按领域划分——营销知识库、经营知识库(定制游/航司)、运营知识库等,各域 Know-how 独立沉淀又可组合调用。
- **Skill 标准化技能**:封装具体业务动作——热点捕捉、文案生成、路线设计、收益测算、价格调整等。
- **Tools 执行接口**:打通各平台/系统 API——抖音/小红书等内容平台、航司系统、PMS/CRM 等业务系统,解决"最后一公里"执行。

#### Layer 2 — 官方专家分身(标准化交付)

利用底座 Harness 能力,组装出"开箱即用"的标准化产品。解决各经营域 80% 的通用需求——是底座的**旗舰应用**与样板间,也是自营业务的载体。

- **营销域分身**:面向中小商家的"超级数字运营"——无需懂底层逻辑,直接带来客流。
- **经营域分身**:面向旅行社/航司的"智能经营助手"——路线设计、收益测算、动态定价。
- 既是**直接营收来源**(自营业务 / SaaS 订阅),又向市场证明底层能力的强大。

#### Layer 3 — 用户自定义分身(生态化)

允许大型文旅集团、旅行社、航司、MCN 等调用底座 Harness 能力,注入**私有认知**(品牌调性、业务逻辑、私域数据),构建专属 Agent。

- **长尾个性化需求**:文旅各细分赛道(定制游、航司、目的地运营等)有极个性化业务逻辑,官方分身无法完全覆盖。
- **Lock-in 粘性效应**:大客户用底座 RAG 与 Tools 搭建专属工作流后,迁移成本极高。
- **反哺底座飞轮**:自定义产生的新玩法、新认知,经提炼后可沉淀到底座,形成生态飞轮。

### 4.3 目标客户与商业模式

双层级架构 + 多能力域,让 Wanpix.ai 在不同客户眼中扮演不同角色:

| 客户类型 | 角色定位 | 售卖内容 | 模式 |
|---|---|---|---|
| 中小文旅商家(景区/酒店) | 营销专家分身提供商 | 标准 SaaS / 服务 | SaaS 订阅 |
| 旅行社 / 定制游公司 | 经营助手分身提供商 | 定制游设计 / 运营 SaaS | SaaS 订阅 |
| 大型文旅集团 / 航司 / MCN | 文旅经营能力的 PaaS 平台 | 底层能力与生态 | PaaS 计量计费 |

#### 对中小文旅商家 — 营销专家分身提供商

以极低成本雇佣一支"永不离职、24h 在线、拥有全行业顶尖营销经验"的超级数字运营团队。

#### 对旅行社 / 航司 — 经营助手分身提供商

把定制游产品经理、收益管理专家的 Know-how 数字化,变成 7×24 在线的智能经营助手,辅助路线设计、收益测算、动态定价等核心经营决策。

#### 对大型文旅集团 / 航司 / MCN — 文旅经营能力的 PaaS 平台

不仅自己造"车"(官方分身),还开放顶级"造车平台",允许客户基于底座构建符合自身业务逻辑的"特种车辆"。

### 4.4 商业壁垒

1. **Know-how 壁垒**:沉淀的多经营域(营销、经营、运营)专家 Know-how 与知识图谱,构成核心护城河。Know-how 越沉淀越厚,后来者越难追赶。
2. **生态壁垒**:Layer 3 自定义分身形成的 Lock-in 效应与飞轮反哺,构成网络效应。
3. **数据壁垒**:数据飞轮让各域分身越来越聪明,形成"使用越多越聪明、越聪明越多人用"的正向循环。
4. **双边市场**:同时服务中小商家(SaaS)和大客户(PaaS),形成双边平台效应。
5. **自营反哺**:自营业务产生的真实 Know-how 持续注入底座,形成平台能力的源头活水。

## 五、技术架构

### 5.1 架构总览

围绕双层级产品架构,技术上由**五大核心模块**配套,形成"感知 → 决策 → 执行 → 反馈 → 进化"的完整闭环。

底层技术架构对各经营域(营销域、经营域、运营域等)一视同仁——**Know-how 内容可以变,但沉淀和编排的方法论不变**。新增一个经营域,本质上是新增一套领域 Knowledge + 领域 Skill,底座架构本身无需改造。

#### 架构原则

1. **能力原子化**:所有能力拆解为可复用的原子单元(Skill / Tool)。
2. **知识领域化**:按经营域划分知识库,可独立沉淀又可组合调用。
3. **知识隔离化**:官方知识与用户私有认知严格分离。
4. **闭环进化化**:每次执行结果反哺底座,持续进化。
5. **多租户化**:全链路租户隔离,支持 SaaS 与 PaaS 双模式。

#### 典型调用流

### 5.2 五大核心模块

#### 模块一:Agent Orchestration Engine(编排引擎)

**角色定位**:底座的"大脑"。解决"如何把 Harness 能力组装起来"的问题。

##### ① Multi-Agent Framework — 多 Agent 协作框架

基于 **Mastra**(TypeScript 原生 Agent 框架)搭建多智能体协作体系。支持 **DAG 有向无环图**工作流编排,让"选题 Agent""文案 Agent""分发 Agent"像流水线一样协同工作;并通过其原生的 Agent Network 与 Workflow 能力,支持顺序、并行、循环等多种协同模式。

##### ② Dynamic Planner — 动态任务规划

底层需要一个强大的 Planner,能够根据用户的自然语言指令,自动拆解任务,并调用底层的 Skills 和 Tools。

示例:输入"帮我策划一个五一假期的景区促销"
- 拆解子任务 1:检索五一文旅热点与平台趋势
- 拆解子任务 2:基于景区知识库生成促销策略
- 拆解子任务 3:生成抖音 / 小红书平台调性的文案
- 拆解子任务 4:调用分发 Tools 发布内容
- 拆解子任务 5:埋点追踪效果,回流至飞轮

##### ③ State Management — 状态管理

文旅营销是**长链路任务**,引擎必须能精准记忆上下文状态,确保在多轮对话和复杂任务执行中不丢失信息。支持:
- 多轮对话上下文持久化
- 长链路任务断点续传
- 多 Agent 间状态共享

---

#### 模块二:RAG 2.0 与向量知识中台

**角色定位**:底座的"记忆库"。**区分经营域 + 官方/私有**的多维知识组织核心。

知识库按两个维度组织:**经营域维度**(营销域 / 经营域 / 运营域)和 **所有权维度**(官方 / 租户私有)。新增一个经营域,就是新增一套 Namespace + 领域 Skill,底座架构完全复用。

##### ① Multi-Domain + Multi-Tenant — 多领域多租户知识隔离

在向量数据库(**Milvus** / **Qdrant**)层面实现二维 Namespace 隔离。

```
┌────────────────────────────────────────────────────────────┐
│                  向量数据库 (Milvus/Qdrant)                  │
│                                                            │
│  ┌─────────────────────────────┐  ┌─────────────────────┐  │
│  │     官方知识 (official)       │  │  租户私有 (tenant_xxx) │  │
│  │                             │  │                     │  │
│  │  ┌─────────────────────────┐│  │  ┌─────────────────┐ │  │
│  │  │ · 营销域知识库           ││  │  │ · 私有营销知识  │ │  │
│  │  │   (爆款文案/平台规则)   ││  │  │   (品牌调性)     │ │  │
│  │  └─────────────────────────┘│  │  └─────────────────┘ │  │
│  │  ┌─────────────────────────┐│  │  ┌─────────────────┐ │  │
│  │  │ · 经营域知识库           ││  │  │ · 私有经营知识  │ │  │
│  │  │   (定制游/航司收益)     ││  │  │   (定价模型)     │ │  │
│  │  └─────────────────────────┘│  │  └─────────────────┘ │  │
│  │  ┌─────────────────────────┐│  │  ┌─────────────────┐ │  │
│  │  │ · 运营域知识库           ││  │  │ · 私有运营知识  │ │  │
│  │  │   (目的地运营/用户增长) ││  │  │   (私域SOP)      │ │  │
│  │  └─────────────────────────┘│  │  └─────────────────┘ │  │
│  └─────────────────────────────┘  └─────────────────────┘  │
└────────────────────────────────────────────────────────────┘
```

- 官方分身调用对应域的官方知识库
- 自定义分身调用"官方领域库 + 租户私有库"(混合检索)
- 不同经营域知识可交叉调用(如营销 + 经营协同决策)

##### ② Hybrid Retrieval — 混合检索策略

文旅行业有大量专有名词(特定景区 IP、非遗名称),纯向量检索易出错。采用三路混合模式:

| 检索方式 | 适用场景 | 示例 |
|---|---|---|
| 向量检索 | 语义相似性 | "五一亲子游" → 匹配家庭套餐文案 |
| BM25 关键词 | 精确实体匹配 | "故宫"、"长城"等专有名词 |
| 知识图谱 | 关联推理 | 景区 → 周边 → 餐饮 → 联动套餐 |

三路结果融合排序后输出,显著提高召回准确率。

##### ③ Live Update Pipeline — 知识动态更新管道

文旅热点变化极快,需搭建**自动化 ETL 管道**:

```
[全网抓取] → [清洗结构化] → [向量化] → [入库] → [生效]
   ↑                                         │
   └─────────── 定时调度 / 事件触发 ──────────┘
```

- 实时抓取全网文旅热点
- 监控平台算法规则变化
- 自动清洗并向量化注入知识库

#### 模块三:Skill / Tools 注册与网关

**角色定位**:底座的"手脚"。实现**能力的原子化封装**。

##### ① Standard Protocol — 标准化 Skill 协议

定义一套标准的 Skill 描述协议(基于 **JSON Schema** 或 **OpenAPI**):

```yaml
# Skill 协议示例
name: hot_topic_script_generator
description: 基于热点话题生成抖音平台脚本
input_schema:
  type: object
  properties:
    topic:
      type: string
    scene_type:
      type: string
      enum: [scenic, hotel, experience]
output_schema:
  type: object
  properties:
    script:
      type: string
    hashtags:
      type: array
execution:
  timeout: 30s
  sandbox: true
```

无论官方"脚本生成 Skill",还是用户上传的"私域客服 Skill",都必须遵循该协议才能被 Agent 调用。

##### ② API Gateway — API 聚合网关

统一管理对抖音、小红书、微信等第三方平台 API 的调用:

- **限流**:防止触发平台风控
- **鉴权**:统一管理多平台凭证
- **异常重试**:避免因某个平台接口波动导致整个 Agent 瘫痪
- **协议转换**:屏蔽各平台 API 差异

##### ③ Sandbox Execution — 沙箱执行环境

当用户自定义分身调用复杂代码或工具时,必须在**隔离的沙箱环境**中执行:

- 资源隔离(CPU / 内存 / 网络)
- 超时机制
- 防止恶意代码或死循环拖垮整个底座平台

---

#### 模块四:Agent Builder(低代码 / 无代码搭建器)

**角色定位**:实现"用户 DIY 专属分身"的前端交互核心。位于分身与大脑之间——它是"造车平台",在分身(车)和编排引擎(引擎)之间做组装。

##### ① Visual Workflow Editor — 可视化工作流编辑器

提供类似 **Coze** / **Dify** 的拖拽式界面。用户可通过连线方式,把底座的 RAG、Skill、Tools 像**搭积木**一样组合起来。

##### ② Prompt IDE — 提示词工程辅助

可视化的 Prompt 调试界面,帮助用户注入自己的"认知"。提供 **Few-shot 少样本学习**模板——用户上传自己品牌的优秀文案作为范例,Agent 就能快速学会这种调性。

##### ③ Simulation Sandbox — 仿真测试环境

在用户发布自定义 Agent 前,提供**仿真沙盒**:

- 用历史数据跑一遍
- 评估其生成内容的质量和合规性
- 输出多维评估报告(相关性、平台调性、合规风险)

---

#### 模块五:Data Flywheel(数据飞轮)

**角色定位**:确保"官方分身越来越聪明,底座越来越强"的进化引擎。

##### ① Full-Chain Tracing — 全链路埋点与追踪

记录 Agent 的每一次:
- **思考链(CoT)**:Agent 的推理过程
- **工具调用**:调用了哪些 Skill / Tools,参数与返回
- **业务结果**:视频播放量、核销率等业务指标

##### ② RLHF Interface — RLHF 人类反馈接口

在官方分身和自定义分身后台,提供便捷的**点赞 / 点踩 / 修改**接口。用户反馈自动回流,用于:
- 微调底层 LLM
- 优化 RAG 检索权重
- 调整 Skill 参数

##### ③ A/B Testing — A/B Testing 框架

支持对不同 Agent 策略进行灰度测试,自动筛选出**转化率最高的营销 SOP**,并将其固化为新的官方 Skill。

##### 飞轮闭环

```
  ┌─────────────────────────────────────────────────┐
  │                                                 ▼
STEP 1            STEP 2            STEP 3            STEP 4            STEP 5
Agent 执行   →    行为埋点      →   业务结果     →   反馈回流     →   沉淀为 Skill
编排+知识+工具    CoT+调用日志     播放量/核销率     RLHF + A/B       反哺底座·进化
  │                                                 ▲
  └────────────────────── 闭环 ─────────────────────┘
```

> 最优 SOP 经 A/B 验证后固化为新的官方 Skill,回到 STEP 1 形成闭环。

## 七、数据与安全

### 7.1 多租户隔离

全链路租户隔离,贯穿数据 / 知识 / Skill 三个层面:

- **数据隔离**:向量数据库 Namespace 级别物理隔离
- **知识隔离**:官方库与私有库严格分离
- **Skill 隔离**:用户上传 Skill 仅在该租户内可见

### 7.2 权限管理

基于 **IAM / RBAC** 模型,实现角色 / 资源 / 动作的细粒度授权:

- 角色定义:平台管理员、租户管理员、运营人员、普通用户
- 资源粒度:分身、Skill、数据、知识库
- 动作粒度:读、写、执行、删除

### 7.3 沙箱安全

用户自定义分身执行复杂代码时,必须在隔离沙箱中运行:

- 资源隔离(CPU / 内存 / 网络配额)
- 超时机制(默认 30s)
- 网络白名单(仅允许调用授权 API)
- 执行日志全量留存

### 7.4 合规与隐私

- 用户数据所有权归属用户,平台不读取私有库内容
- 第三方平台 API 调用严格遵守各平台 ToS
- 数据飞轮回流前需进行脱敏处理
- 用户反馈数据用于模型优化前需获得授权

## 八、实施路径与里程碑

整体策略:**营销域切入,经营域纵深,Know-how 平台化是终局**。先通过营销域验证平台能力,再逐步向经营域(定制游、航司等)扩展,最终沉淀为文旅经营域的能力基建平台。

### 8.1 阶段划分

#### Phase 1:底座 MVP(营销域官方分身跑通)

**目标**:验证底座能力,跑通第一个经营域(营销域)的官方专家分身。

- 完成 ① 编排引擎核心(Mastra 集成 + Planner + State)
- 完成 ② RAG 2.0 营销域官方知识库建设(向量库 + 混合检索)
- 完成 ③ 营销域官方 Skill 集(脚本生成、热点捕捉、套餐设计等 5–10 个核心 Skill)
- 完成 ③ 抖音 / 小红书 API 网关对接
- 上线 1–2 个官方专家分身(如景区营销分身)

#### Phase 2:飞轮闭环 + 营销域 SaaS 商业化

**目标**:数据飞轮闭环,中小商家 SaaS 模式跑通,验证"Know-how 沉淀→规模化"逻辑。

- 完成 ⑤ 数据飞轮核心(埋点 + RLHF 接口)
- 完成 A/B Testing 框架
- 验证"执行 → 反馈 → 沉淀 Skill"闭环
- 营销域分身规模化推广,验证 SaaS 商业模式
- 引入计费体系(Billing)

#### Phase 3:经营域扩展(第二、第三域落地)

**目标**:验证平台的多域扩展能力,进入经营域(旅行社定制游 / 航司收益管理)。

- 新增经营域知识库(定制游路线设计、航司收益模型等)
- 新增经营域 Skill 集(路线测算、收益定价、舱位调度等)
- 上线经营域官方分身(如定制游产品经理分身)
- 打通经营域业务系统 API(PMS / 航司系统 / CRM)
- 验证"新增一个经营域 = 新增一套领域 Knowledge + Skill"的扩展效率

#### Phase 4:Agent Builder + PaaS 开放

**目标**:开放 PaaS 能力,吸引大客户构建自定义分身,从"自己造车"扩展到"开放造车平台"。

- 完成 ④ Agent Builder(Visual Editor + Prompt IDE + Simulation)
- 完善多租户隔离与权限管理(支持多域混合调用)
- 完成沙箱执行环境
- 引入首批 PaaS 大客户(大型文旅集团 / 航司 / MCN)
- 验证 Lock-in 效应与生态飞轮反哺

#### Phase 5:生态繁荣 + 跨域扩展

**目标**:形成双边平台效应,生态飞轮自驱运转,Know-how 平台价值最大化。

- 开放 Skill 市场,引入第三方 Skill 开发者
- 自定义分身反哺底座,各经营域知识相互促进
- 跨行业扩展(从文旅延伸到其他垂直领域)
- 平台从"文旅经营平台"升级为"垂直行业 Know-how 基建"

### 8.2 关键里程碑

| 里程碑 | 标志 |
|---|---|
| M1 · 底座可用 | 营销域官方分身能完成端到端营销任务 |
| M2 · 飞轮闭环 | 飞轮反哺产生第一个新官方 Skill |
| M3 · SaaS 验证 | 中小商家付费率与留存率达预期 |
| M4 · 经营域首战 | 经营域(定制游/航司)官方分身上线并验证 |
| M5 · PaaS 开放 | 首个大客户上线自定义分身 |
| M6 · 多域协同 | 跨域(营销+经营)协同决策跑通 |
| M7 · 生态繁荣 | 第三方 Skill 开发者入驻 |

## 九、风险与应对

### 9.1 技术风险

| 风险 | 影响 | 应对 |
|---|---|---|
| LLM 幻觉导致营销/经营内容失真 | 信任度下降 | RAG 强约束 + A/B 验证 + 人工审核机制 |
| 第三方平台/系统 API 风控或波动 | 分发与执行能力受限 | 多账号轮换 + 限流策略 + 平台合规对接 + 异常重试 |
| 多 Agent 协作复杂度高 | 系统不稳定 | DAG 工作流可视化调试 + 状态追踪 |
| 知识库更新滞后 | 业务内容过时 | 自动化 ETL 管道 + 热点/规则实时监控 |
| 多域协同推理复杂度高 | 跨域决策质量下降 | 分域独立验证 + 跨域编排测试 + 人工兜底 |

### 9.2 业务风险

| 风险 | 影响 | 应对 |
|---|---|---|
| 中小商家付费意愿低 | SaaS 收入不及预期 | 免费试用 + 效果分成模式 + 标杆案例 |
| 大客户自建团队 | PaaS 客户流失 | 强调知识壁垒 + Lock-in 效应 + 生态价值 |
| 经营域扩展速度不及预期 | 第二曲线增长滞后 | 选择高价值细分赛道(如定制游)集中突破 |
| 行业竞争加剧 | 市场份额受挤压 | 加速飞轮闭环,构建数据壁垒与 Know-how 壁垒 |

### 9.3 数据风险

| 风险 | 影响 | 应对 |
|---|---|---|
| 用户私有数据泄露 | 法律与信任风险 | 多租户严格隔离 + 沙箱执行 + 数据脱敏 |
| 数据飞轮数据质量差 | 模型进化方向偏离 | 数据清洗管道 + 人工标注 + 异常检测 |
| 经营域数据敏感性高 | 合规风险 | 按域分级权限 + 审计日志 + 合规审查机制 |

## 十、总结

Wanpix.ai 的本质,是把**文旅经营域的专家 Know-how** 做成可规模化复制的**数字化分身**——这不仅是 SaaS 工具,更是文旅经营的**能力基建**。

核心理念:**把自己的 Know-how 做成平台的核心竞争力。**

平台既可以自己造车(做官方分身、做自营业务),也可以开放造车能力(做 PaaS、让客户自己造车)——两者相互促进,形成双向飞轮。

本方案的核心价值:

1. **产品侧**:从营销域切入,逐步扩展到经营域(定制游、航司等)乃至全经营链路;双层级架构覆盖中小商家(SaaS)与大客户(PaaS)双边市场,商业天花板从工具跃升到生态平台。
2. **技术侧**:五大核心模块形成"感知 → 决策 → 执行 → 反馈 → 进化"完整闭环;多领域知识库架构支持"新增一个经营域 = 新增一套 Knowledge + Skill"的高效扩展,数据飞轮构建持续进化的能力底座。
3. **壁垒侧**:Know-how + 生态 + 数据 + 双边市场 + 自营反哺五重壁垒,后来者难以追赶;且 Know-how 越沉淀越厚,飞轮越转越快,壁垒随时间递增。

核心信念:**不仅自己造"车",还开放顶级"造车平台"**——把 Know-how 做成平台,把平台做成生态,把生态做成行业基建。这在 ToB 业务中是一个极具想象力和壁垒的商业模式。
